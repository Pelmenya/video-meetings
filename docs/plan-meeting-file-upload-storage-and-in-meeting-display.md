# Plan: Загрузка файла встречи для хранения, обработки и отображения внутри встречи

**PRD:** docs/prd-meeting-file-upload-storage-and-in-meeting-display.md
**Дата:** 2026-07-09

## Технические решения (уточнены у пользователя, не описаны в PRD)

- **Хранилище файлов:** локальный диск (директория на хосте `apps/api`, путь через env-переменную, gitignored), без S3/MinIO.
- **Асинхронная обработка:** без очереди/Redis — статус хранится в Postgres (`MeetingFile.status`), обрабатывается внутренним воркером на `setInterval` в самом Nest-процессе.
- **Транскрипция:** внешний API (Whisper API или аналог) по ключу из env, без self-hosted модели.
- **Транскодирование:** ffmpeg (`fluent-ffmpeg` + `ffmpeg-static` или системный ffmpeg) — стандартный выбор, не требовал уточнения.
- Отдельной страницы встречи (`/meetings/[id]`) в проекте пока нет — её создание включено в Фазу 2 как необходимая основа для отображения списка файлов и плеера, без которой ни один критерий готовности PRD не проверить.

## Фазы реализации

### Фаза 1: Backend — загрузка, хранение и выдача файлов (tracer bullet, backend)

**Цель:** Участник встречи может через API загрузить файл (запись или вложение), файл надёжно сохраняется на диске и связывается с встречей, доступен списком и потоковой отдачей. Статус на этом этапе всегда сразу `READY` (реальный жизненный цикл статусов появится в Фазе 3) — это осознанное упрощение, чтобы дать сквозной путь для Фазы 2.
**Затрагивает:** backend, database
**Задачи:**

- [x] Prisma: модель `MeetingFile` (meetingId, uploaderId, kind: RECORDING | ATTACHMENT, originalName, mimeType, sizeBytes, storagePath, status: QUEUED | PROCESSING | READY | ERROR, errorMessage, createdAt/updatedAt) + миграция.
- [x] Модуль `files` по CQRS-конвенции проекта: `UploadMeetingFileCommand`/handler (сохраняет файл на диск через multer, создаёт запись в БД), `GetMeetingFilesQuery`/handler (список файлов встречи), маппер ответа.
- [x] Эндпоинты под `JwtAuthGuard`, авторизация через `assertHostOrParticipant`: `POST /meetings/:id/files` (multipart upload), `GET /meetings/:id/files` (список), `GET /meetings/:id/files/:fileId/content` (потоковая отдача с поддержкой `Range` для видео/аудио).
- [x] Валидация загрузки: allow-list MIME-типов, лимит размера файла, отклонение с понятным русскоязычным сообщением (`400`) при нарушении.
- [x] Тесты: unit-тесты handler'ов (в т.ч. отказ доступа не-участнику встречи → 404, отказ по типу/размеру), e2e-тест полного цикла upload → list → content.

**Когда готова:** Через API можно загрузить файл на встречу (только её участником), получить список файлов встречи и получить содержимое загруженного файла обратно; недопустимый тип/размер отклоняется с понятным сообщением.

### Фаза 2: Frontend — страница встречи с загрузкой и списком файлов (tracer bullet, frontend)

**Цель:** Участник открывает встречу в интерфейсе, загружает файл через UI и видит его в списке файлов встречи со статусом.
**Затрагивает:** frontend
**Задачи:**

- [ ] Новый роут `apps/web/src/app/meetings/[id]/page.tsx` — базовый экран встречи (заголовок, дата, статус, список участников), auth-gating по образцу `page.tsx` (localStorage-токен, редирект на `/login`).
- [ ] `lib/api.ts`: клиентские функции `uploadMeetingFile`, `getMeetingFiles`, построение URL для `content`-эндпоинта.
- [ ] Компонент загрузки файла: выбор файла, индикация прогресса, отображение ошибки (сеть/формат/размер из `ApiError.messages`) с возможностью повторить попытку.
- [ ] Компонент списка файлов встречи: имя, тип, статус-`Chip` (по аналогии с `STATUS_LABELS`/`STATUS_COLORS` для встреч), периодический опрос (`polling`) для обновления статуса; на главной странице проставить `href`/`onPress` для перехода со списка встреч на `/meetings/[id]`.
- [ ] Визуальная проверка через Playwright MCP (светлая и тёмная тема) + проверка через skill `ui-ux-pro-max`; отдельного unit-тестового набора для `apps/web` в проекте нет — это единственный способ верификации фазы.

**Когда готова:** Участник может открыть встречу, загрузить видео/аудио или произвольное вложение через UI, увидеть его в списке файлов со статусом; ошибка загрузки показывается понятным сообщением с возможностью повтора.

### Фаза 3: Backend — асинхронное транскодирование и жизненный цикл статуса

**Цель:** Загруженные аудио/видео файлы проходят реальный цикл `QUEUED → PROCESSING → READY/ERROR`, transcode в единый формат хранения/воспроизведения, и не отдаются на воспроизведение до готовности.
**Затрагивает:** backend
**Задачи:**

- [ ] Добавить ffmpeg-зависимость (`fluent-ffmpeg` + `ffmpeg-static`) в `apps/api`.
- [ ] Сервис-воркер (`setInterval` в рамках Nest-процесса): выбирает файлы `kind=RECORDING` со статусом `QUEUED`, помечает `PROCESSING`, транскодирует в единый формат, при успехе сохраняет `processedPath` и статус `READY`, при ошибке — `ERROR` + `errorMessage`.
- [ ] Обновить `UploadMeetingFileCommand`: для `RECORDING` — статус `QUEUED` при создании (вместо заглушки из Фазы 1); для `ATTACHMENT` — по-прежнему сразу `READY` (транскодирование не требуется).
- [ ] Обновить эндпоинт `GET /meetings/:id/files/:fileId/content`: для `RECORDING` отдавать `processedPath` только при статусе `READY`, иначе — явный отказ (файл ещё не готов к воспроизведению).
- [ ] Тесты: unit-тест перехода статусов воркера (с моком ffmpeg-вызова), e2e-тест «файл дошёл до `READY` и стал доступен по content-эндпоинту», тест ошибки обработки (битый файл → `ERROR` с причиной, контент недоступен).

**Когда готова:** Загруженная запись встречи проходит через статусы очереди/обработки и становится воспроизводимой только после успешного транскодирования в единый формат; ошибка обработки отражается статусом `ERROR` с причиной.

### Фаза 4: Frontend — встроенный плеер, превью и живой статус

**Цель:** Участник воспроизводит видео/аудио и просматривает прочие поддерживаемые файлы прямо в интерфейсе встречи, без скачивания, видя актуальный статус обработки.
**Затрагивает:** frontend
**Задачи:**

- [ ] Компонент плеера (`<video>`/`<audio>`) на content-эндпоинт из Фазы 1/3, активен только при статусе `READY`.
- [ ] Компонент превью для не-медиа типов (изображения — инлайн `<img>`, PDF — `<iframe>`/`<embed>`; для неподдерживаемых типов превью — понятное сообщение «предпросмотр недоступен» без блокировки остального интерфейса).
- [ ] Отображение состояний `QUEUED`/`PROCESSING`/`ERROR` в списке файлов (плеер/превью скрыты, пока не `READY`; ошибка — с текстом причины).
- [ ] Обновить polling из Фазы 2, чтобы он отражал реальные переходы статуса из Фазы 3.
- [ ] Визуальная проверка через Playwright MCP (светлая/тёмная тема) для всех состояний (в очереди, обработка, готово, ошибка) + `ui-ux-pro-max`.

**Когда готова:** Видео/аудио воспроизводится встроенным плеером без скачивания; файлы поддерживаемых прочих типов открываются встроенным превью без скачивания; статус обработки в UI соответствует реальному состоянию файла.

### Фаза 5: Backend — транскрипция записей

**Цель:** После успешного транскодирования аудио/видео файл автоматически отправляется во внешний API транскрипции, результат сохраняется и становится доступен по API.
**Затрагивает:** backend, database
**Задачи:**

- [ ] Prisma: модель `Transcript` (meetingFileId — уникальная связь 1:1 с `MeetingFile`, status: PENDING | PROCESSING | READY | ERROR, text, errorMessage, createdAt/updatedAt) + миграция.
- [ ] Расширить воркер из Фазы 3: после успешного `READY` транскодирования для `RECORDING`-файла — шаг отправки во внешний API транскрипции, сохранение результата в `Transcript`; ошибка транскрипции не блокирует уже готовое воспроизведение файла (независимый статус).
- [ ] Конфигурация внешнего API: ключ через env (`apps/api/.env.example` обновить), таймауты и обработка ошибок вызова с понятным `errorMessage`.
- [ ] Эндпоинт `GET /meetings/:id/files/:fileId/transcript` (под `JwtAuthGuard` + `assertHostOrParticipant`): текст расшифровки, если готова, статус — иначе.
- [ ] Тесты: unit-тест шага транскрипции воркера (мок внешнего API), e2e-тест «после обработки записи транскрипт становится доступен», тест ошибки вызова API (транскрипт `ERROR`, файл остаётся `READY`/воспроизводимым).

**Когда готова:** После завершения обработки аудио/видеозаписи через её эндпоинт транскрипта доступна текстовая расшифровка; сбой транскрипции не влияет на доступность самого файла для воспроизведения.

### Фаза 6: Frontend — отображение транскрипции

**Цель:** Участник видит текстовую расшифровку рядом с записью прямо в интерфейсе встречи, как только она готова.
**Затрагивает:** frontend
**Задачи:**

- [ ] `lib/api.ts`: функция `getMeetingFileTranscript`.
- [ ] Компонент панели транскрипции рядом с плеером для файлов типа `RECORDING`, с состояниями «готовится» / «недоступна (ошибка)» / текст расшифровки, обновление через polling.
- [ ] Встраивание панели в экран встречи (`apps/web/src/app/meetings/[id]/page.tsx`) рядом с плеером.
- [ ] Визуальная проверка через Playwright MCP (светлая/тёмная тема, все состояния транскрипции) + `ui-ux-pro-max`.

**Когда готова:** Для загруженной и обработанной записи встречи участник видит связанную текстовую расшифровку прямо в интерфейсе встречи без скачивания каких-либо файлов.
