## 1. Multipart upload: `@nestjs/platform-express` + multer `diskStorage`

**Механизм.** `@nestjs/platform-express` реэкспортирует `multer` (v1.x на Express 4/5 через `@nestjs/platform-express` ^11 — Nest сам пинит совместимую версию в своих peer-зависимостях, отдельно ставить `multer` не нужно, но полезно свериться, что `npm ls multer` даёт одну версию, а не дублирует её в `apps/api/node_modules` из-за workspace-hoisting — в проекте уже есть прецедент такого сорта проблемы с `@nestjs/platform-express`, см. `CLAUDE.md` корня). Используем `FileInterceptor('file', options)` на `POST /meetings/:id/files` (один файл на запрос — под сценарии PRD этого достаточно, `FilesInterceptor` не нужен).

**Хранение и имя файла.** `diskStorage({ destination, filename })`:

- `destination` — **не** `express.static`-раздаваемая директория (см. секцию 7); абсолютный путь из env (`MEETING_FILES_STORAGE_PATH` или аналог), с `fs.mkdirSync(dest, { recursive: true })` на старте модуля/при первой записи.
- `filename` — **никогда** не использовать `file.originalname` напрямую ни в имени, ни в подстроке пути (классический path traversal через `../../etc/passwd`-подобные имена или `null`-байты). Практика: генерировать имя сервером целиком — `${randomUUID()}${extname(file.originalname)}` (расширение можно оставить только для читаемости/дебага, оно не участвует в определении типа — см. секцию 1.2), сам `originalName` сохраняется отдельно как поле в БД (`MeetingFile.originalName`), а не как часть пути на диске. Так коллизии имён и directory traversal закрываются одним приёмом — на диске лежит `<uuid>.<ext>` в плоской или `meetingId`-партиционированной директории, и то и то безопасно, но партиционирование по `meetingId` (тоже UUID, генерируемый Prisma) упрощает ручной аудит на проде.

**MIME-валидация — двухступенчатая, ни в коем случае не одноступенчатая.**

1. `fileFilter` в `MulterOptions` (или `ParseFilePipe`/`ParseFilePipeBuilder` с `FileTypeValidator`) — это **первый, дешёвый и обязательно недостаточный** фильтр. `FileTypeValidator` в NestJS сверяет `file.mimetype`, который приходит из заголовка `Content-Type` конкретной части `multipart/form-data` — то есть это то, что **сообщил клиент**, а не то, что реально записано в байтах файла. Официальная документация Nest прямо признаёт это ограничение и рекомендует кастомную проверку для более надёжного сценария.
2. **Контент-сниффинг после записи на диск (или из буфера в памяти до записи, если объём позволяет)** — пакет `file-type` (ESM-only начиная с v17+, актуальная версия на 2025/2026 — v19/v20; при `"type": "commonjs"` в `apps/api/package.json` подключать через динамический `await import('file-type')`, а не `require` — типичная ловушка). `file-type` читает первые несколько десятков/сотен байт (magic bytes) и возвращает реальный тип. Проверка встраивается в `UploadMeetingFileHandler` **после** сохранения multer-ом временного файла, **до** записи `READY`/создания финальной БД-записи: если сигнатура не совпадает с allow-list (или тип неопределим для типов, которые должны быть строго бинарными, например видео/аудио) — удалить файл с диска и бросить `BadRequestException` с русским сообщением, как остальные валидации в проекте (см. `class-validator`-конвенцию в `apps/api/CLAUDE.md`).

**Лимит размера.** Multer `limits.fileSize` (байты) — обязателен, иначе Express/multer примут файл любого размера пока не кончится диск. Отдельно от multer — глобальный body size limit Express: Nest 11 на Express-платформе по умолчанию использует `body-parser`/встроенный Express JSON/urlencoded парсер с лимитом ~100kb для JSON/urlencoded, но это **не относится** к `multipart/form-data`, которую целиком разбирает multer/busboy в потоковом режиме — так что для файлового аплоада именно multer `limits.fileSize` — единственный релевантный лимит, JSON body-parser limit трогать не нужно. При превышении multer кидает `MulterError` с кодом `LIMIT_FILE_SIZE`, который в Nest долетает как `PayloadTooLargeException` только если явно обработан — стоит обернуть в exception filter или проверить в handler'е `err.code === 'LIMIT_FILE_SIZE'` → 400 с понятным русским текстом (PRD явно требует "понятное сообщение" при превышении лимита).

**Рекомендация:**
```
@nestjs/platform-express (уже в зависимостях, версия синхронна с @nestjs/core ^11)
file-type ^19 или ^20 (ESM — динамический import)
```
Multer `diskStorage`: `destination` = env-путь вне static root, `filename` = `randomUUID() + extname`. Порядок валидации: (1) multer `fileFilter` по расширению/заявленному mimetype как быстрый предфильтр + `limits.fileSize`, (2) после физической записи — `file-type` по реальным байтам, при несовпадении с allow-list — удалить файл и 400. `originalName` — только метаданные в БД, никогда не часть пути на диске.

## 2. Отдача файла с поддержкой HTTP `Range` (стриминг видео/аудио)

**Готовых решений в самом Nest нет.** `StreamableFile` в текущей версии Nest — низкоуровневая обёртка вокруг `Readable`-потока для *полной* отдачи файла; она не парсит `Range`, не выставляет `206`/`Content-Range` сама. Есть открытая и подтверждённая проблема (`nestjs/nest#14873`): при `Range`-запросах от iOS Safari на аудио через `StreamableFile` сервер может упасть с необработанной ошибкой `Premature close`/`body.errorLogger is not a function` при обрыве потока клиентом (типичное поведение iOS — открывает и почти сразу обрывает соединение при probe-запросе). На момент этого исследования фикса от Nest нет — вывод: **не полагаться на голый `StreamableFile` для range-эндпоинта**, а либо руками валидировать это на реальном iOS-устройстве/симуляторе, либо оборачивать поток в собственный error-handler.

**Рекомендуемый паттерн — ручной, через `@Res({ passthrough: true })`:**

1. Проверить заголовок `Range` (`@Headers('range') range?: string`). Если его нет — отдать файл целиком (можно через `StreamableFile` с `Content-Type`/`Content-Length` из `fs.statSync`, без `Range`-логики).
2. Если `Range` есть — распарсить через пакет `range-parser` (маленькая, стабильная библиотека, которую использует сам Express внутри `send`/`serve-static`; можно class добавить явной зависимостью, а не полагаться на транзитивную). Обработать погранично: `-1`/`-2` от `range-parser` → `416 Range Not Satisfiable` с заголовком `Content-Range: bytes */<size>`. Поддерживать **только один диапазон** (multi-range для `<video>`/`<audio>` браузерам не нужен — они всегда шлют один `Range: bytes=start-` или `start-end`).
3. Выставить вручную: `response.status(206)`, `Content-Range: bytes ${start}-${end}/${fileSize}`, `Accept-Ranges: bytes`, `Content-Length: ${end - start + 1}`, `Content-Type` = сохранённый в БД `mimeType` файла (не заново детектить на каждый запрос).
4. Стримить `fs.createReadStream(path, { start, end })` через `stream.pipe(response)`, обязательно с обработчиком `stream.on('error', ...)` — не давать необработанной ошибке потока уронить процесс (это и есть источник бага из issue выше). При обрыве клиентом (`ECONNRESET`/`premature close`) — тихо завершать, не логировать как критическую ошибку.
5. `GET .../content` для файла со статусом не-`READY` (после Фазы 3) — `404`/`409` с понятным сообщением, до Range-логики.

**Рекомендация:** ручная реализация поверх `range-parser` (`npm i range-parser` + `@types/range-parser` — обе крошечные, используются самим Express/`serve-static` внутри) и `fs.createReadStream`, не `StreamableFile` для файлов, которые реально скрабятся (RECORDING). Для не-стримируемых типов (ATTACHMENT: PDF/изображения) `StreamableFile` без Range-логики достаточен — `<iframe>`/`<img>` не делают partial requests так агрессивно, как `<video>`/`<audio>`, но `Accept-Ranges: bytes` всё равно стоит проставлять единообразно на весь content-эндпоинт, раз уж пишем ручную реализацию — код один и тот же для обоих kind.

## 3. In-process polling worker без очереди

**Идиома.** `@nestjs/schedule` (`^4.x`, совместим с Nest 11 — не в зависимостях `apps/api` на момент этого исследования, добавить) даёт `@Interval(ms)` — декларативнее и тестируемее голого `setInterval` в `OnModuleInit`, плюс `SchedulerRegistry` умеет `deleteInterval`/`getInterval` для управляемой остановки. План уже называет `setInterval` — оба варианта реализуют одно и то же поверх Node таймеров; `@nestjs/schedule` просто снимает бойлерплейт регистрации/очистки и даёт единообразный способ тестировать (мокать `SchedulerRegistry`), поэтому предпочтителен.

**Питфоллы, которые нужно закрыть явно:**

- **Наложение запусков (overlapping runs).** Если обработка файла (транскодирование) длится дольше интервала опроса, следующий тик `@Interval` стартует, пока предыдущий ещё работает → два воркер-прохода одновременно читают/пишут одни и те же `QUEUED`-строки. Нужен in-memory guard: булев флаг/`Promise` "уже выполняется" на уровне сервиса (`private isRunning = false`), тик, заставший `isRunning === true`, выходит немедленно. Это защищает только *в рамках одного процесса* — см. следующий пункт.
- **Однопроцессность — явное архитектурное ограничение, которое стоит зафиксировать как осознанный компромисс, а не забытый риск.** In-memory guard решает overlapping только пока `apps/api` работает в единственном экземпляре. Если в будущем появится второй под/реплика (даже просто `pm2 cluster` с >1 воркером или два контейнера за балансировщиком) — оба процесса независимо опрашивают одну и ту же таблицу и без БД-уровня блокировки возьмут одну и ту же `QUEUED`-строку в обработку параллельно (race). Раз очереди/Redis сознательно нет, **это ограничение стоит явно записать в риски** (см. финальную секцию) и держать `apps/api` в единственном экземпляре, пока это не изменится намеренно.
- **Безопасный claim даже в рамках одного процесса — задел на будущее и просто хорошая практика.** Вместо `findMany({ status: 'QUEUED' })` + отдельный `update` (racy даже intra-process при двух почти одновременных тиках, если guard забыт/сломан) — атомарный claim одним запросом:
  - Простой вариант, без сырого SQL: `prisma.meetingFile.updateMany({ where: { id: candidate.id, status: 'QUEUED' }, data: { status: 'PROCESSING' } })` и проверка `count === 1` — оптимистическая блокировка через условие в `WHERE`, конкурентный второй вызов получит `count === 0` и просто пропустит строку. Этого достаточно для однопроцессного воркера.
  - Более строгий вариант (задел на много процессов) — `SELECT ... FOR UPDATE SKIP LOCKED` через `prisma.$queryRaw` внутри `prisma.$transaction`, который блокирует и сразу пропускает уже залоченные другим воркером строки на уровне Postgres, а не на уровне приложения. Prisma не имеет нативного API для `FOR UPDATE SKIP LOCKED` (открытый issue `prisma/prisma#5983`, не решён), так что это всегда `$queryRaw`/`$executeRaw`. Для текущего скоупа (один процесс) — избыточно, но если решение будет пересмотрено в сторону нескольких реплик, это первая точка доработки.
- **Graceful shutdown.** `OnModuleDestroy` (или `beforeApplicationShutdown`) должен: (1) прекратить брать новые `QUEUED`-строки (снять интервал через `SchedulerRegistry.deleteInterval`/выставить флаг "останавливаемся"), (2) по возможности дождаться завершения текущей обработки с таймаутом, иначе на `SIGTERM` (например, при деплое/рестарте) файл останется зависшим в `PROCESSING` навсегда — при старте воркера стоит также предусмотреть "reap" шаг: строки, застрявшие в `PROCESSING` дольше разумного таймаута (например, дольше ожидаемого времени транскодирования конкретного размера файла), переводить обратно в `QUEUED` или в `ERROR` с сообщением "processing interrupted" — иначе рестарт процесса посреди обработки файла оставляет файл в вечном "обрабатывается", и PRD-критерий "статус отражает реальное состояние" нарушается. Необходимо явно вызвать `app.enableShutdownHooks()` в `main.ts` — без этого `OnModuleDestroy` в Nest вообще не вызывается на `SIGTERM`.

**Рекомендация:**
```
@nestjs/schedule (совместимая с Nest 11 версия — ^4.x)
```
`@Interval(POLL_MS)` + `isRunning`-guard в сервисе воркера, claim через `updateMany({ where: { status: 'QUEUED' }, data: { status: 'PROCESSING' } })` с проверкой `count`, `app.enableShutdownHooks()` в `main.ts` + `OnModuleDestroy` останавливает интервал и (по возможности) ждёт текущий job, плюс отдельный «reap» проход при старте приложения для файлов, зависших в `PROCESSING`. В коде и/или в `apps/api/CLAUDE.md` явно зафиксировать: воркер предполагает единственный экземпляр `apps/api`.

## 4. ffmpeg-транскодирование

**Важная находка, меняющая исходную формулировку плана.** Плана называет `fluent-ffmpeg` + `ffmpeg-static` как "стандартный выбор, не требовал уточнения" — но **репозиторий `fluent-ffmpeg/node-fluent-ffmpeg` был архивирован автором 22 мая 2025 года** (read-only, issues/PR не принимаются) с явным заявлением, что пакет "больше не работает корректно с последними версиями ffmpeg" и не будет чиниться под будущие edge-cases. При этом пакет по-прежнему имеет ~2M скачиваний/неделю на npm и продолжает работать для типовых сценариев (transcode в MP4/H.264+AAC — ровно то, что нужно здесь, не экзотика) — то есть использовать его **можно**, но статус нужно знать и заложить план Б.

- `ffmpeg-static` (сам бинарник) — в отличие от обёртки, активно поддерживается независимо (последний релиз на момент исследования — конец 2025), не завязан на архивацию `fluent-ffmpeg`, работает как самостоятельный пакет "скачай статический ffmpeg-бинарь под текущую ОС/архитектуру при `npm install`".
- Форк `@ts-ffmpeg/fluent-ffmpeg` (типизированный форк) — тоже помечен как неподдерживаемый, не решает проблему.
- **Практический вывод:** для скоупа задачи (один конвейер: video/audio → единый MP4 H.264/AAC) `fluent-ffmpeg` + `ffmpeg-static` — рабочий и самый быстрый путь к результату (минимум кода, знакомый fluent API), но стоит закладывать, что при первом серьёзном edge-case (специфичный кодек на входе, странный контейнер) патча от апстрима не будет — тогда переход на прямой вызов `ffmpeg-static`'ного бинарника через `child_process.spawn(ffmpegPath, args)` без обёртки. Обёртка `fluent-ffmpeg` даёт удобный `.on('progress')`/`.on('error')`/`.on('end')` API поверх того же `spawn`, так что миграция в случае проблем — не полная переработка, а замена одного модуля.

**Указание бинарника.** `ffmpeg-static` экспортирует путь к бинарнику как default export (`import ffmpegPath from 'ffmpeg-static'`), который передаётся в `fluent-ffmpeg.setFfmpegPath(ffmpegPath)` один раз при старте модуля (например, в конструкторе воркер-сервиса или в `main.ts`). На Windows dev-окружении (см. `env.OS Version: Windows 10` в этом репозитории) бинарник — `.exe`, `ffmpeg-static` резолвит платформу автоматически через `postinstall`; в Docker/Linux-проде — соответствующий Linux-бинарь, тоже автоматически по `process.platform`/`process.arch` на момент установки (важно: `npm install` должен выполняться **на целевой платформе** — если образ билдится на другой архитектуре без platform-эмуляции, бинарь скачается не тот; для Docker-мультистейдж-сборки — `npm ci` должен идти уже в целевом (Linux amd64/arm64) слое, не копироваться с хоста).

**Формат вывода.** MP4-контейнер, видеокодек H.264 (`libx264`), аудиокодек AAC — самый широко поддерживаемый браузерами комбо на 2025/2026 (Chrome, Firefox, Safari, Edge — все поддерживают H.264+AAC в MP4 без доп. фолбэков; WebM/VP9 шире по некоторым метрикам открытости кодека, но требует MP4-фолбэка для Safari, что усложняет "единый формат хранения" из требования PRD — поэтому один MP4/H.264/AAC действительно закрывает требование "единый формат" без дублирования файлов на диске). Для чисто аудио-записей (без видеодорожки) — тот же принцип, но контейнер можно оставить MP4/AAC (`.m4a`) или отдельно решить на этапе реализации, если понадобится более компактный аудио-формат (не блокирует это исследование).

**Progress/ошибки.** `fluent-ffmpeg`: `.on('progress', (p) => ...)` даёт `percent`/`timemark` (не всегда точный `percent` для всех источников — не полагаться на него как на единственный сигнал "жив ли процесс", учитывать также общий таймаут на job), `.on('error', (err, stdout, stderr) => ...)` — здесь `stderr` от ffmpeg обычно содержит содержательную причину сбоя (битый файл, неподдерживаемый кодек на входе) — это то, что стоит класть (обрезанным) в `MeetingFile.errorMessage`, не сырое `err.message`, которое часто просто "ffmpeg exited with code 1".

**Temp-файл + atomic rename.** Транскодировать всегда в путь вида `<processedPath>.tmp-<uuid>` в той же директории (важно — в той же файловой системе/разделе, что и финальный путь, иначе `fs.rename` не будет атомарным и упадёт в `EXDEV`, тихо деградировав в copy+delete на некоторых платформах), и только после успешного завершения ffmpeg-процесса (`.on('end')`) — `fs.promises.rename(tmpPath, finalPath)`, затем обновление `status: 'READY'` в БД в той же логической операции (порядок: rename → DB update, не наоборот — иначе есть окно, где статус уже `READY`, а файла ещё физически нет по этому пути). Если процесс воркера падает между rename и DB-update — при рестарте reap-логика из секции 3 должна такие файлы просто пере-обработать (или как минимум не выдавать их как READY, если статус в БД не подтверждён), сам `.tmp-*`-файл нужно чистить (по возрасту, отдельным периодическим шагом, вне скоупа этого фазового плана, но стоит зафиксировать как техдолг).

**Рекомендация:**
```
fluent-ffmpeg ^2.1.x (архивирован upstream, но рабочий; план Б — прямой child_process.spawn на ffmpeg-static бинарнике)
ffmpeg-static ^5.x (активно поддерживается отдельно)
@types/fluent-ffmpeg (dev)
```
`setFfmpegPath(ffmpegPathFromStatic)` один раз при инициализации сервиса, выход — MP4/H.264/AAC, пишем в `*.tmp-<uuid>` в целевой директории, `rename` на успех, `errorMessage` — из `stderr` в `.on('error')`, а не голый `err.message`.

## 5. Внешняя транскрипция (Whisper API)

**Жёсткий лимит 25MB на файл** — актуален и на 2026 год для `whisper-1` и для более новых `gpt-4o-transcribe`/`gpt-4o-mini-transcribe` на том же `/audio/transcriptions`-эндпоинте (лимит не менялся). Это первично важно для этой фичи: запись встречи, особенно видео, почти гарантированно превысит 25MB даже на несколько минут разговора.

**Что это значит для шага в Фазе 5 воркера:**

1. **Извлекать только аудиодорожку перед отправкой**, а не слать видеофайл целиком — тем же ffmpeg-проходом (или отдельным быстрым шагом сразу после транскодирования в Фазе 3, раз файл уже открыт ffmpeg-ом) вырезать аудио в компактном кодеке (AAC/MP3, 64–128kbps моно/стерео речи достаточно) — это резко снижает вероятность превышения лимита: 25MB на MP3 64–128kbps — это по грубой оценке около 20+ минут, чего может не хватить на длинную встречу, но кратно лучше, чем видеофайл.
2. **Если аудиодорожки всё равно недостаточно (длинные встречи)** — chunking: разбивать аудио ffmpeg-ом на сегменты ≤25MB (по длительности, с оценкой битрейта) и звать API транскрипции на каждый чанк отдельно, затем конкатенировать текстовые результаты в порядке чанков. Разбиение "по тишине" (VAD) даёт более чистые границы фраз, чем разбиение по фиксированной длительности, но фиксированная длительность (например, по 10–15 минут с небольшим нахлёстом) — значительно проще в реализации и достаточна для скоупа этой фичи; VAD-разбиение можно оставить как улучшение вне текущего плана.
3. **Таймауты/ретраи.** Внешний HTTP-вызов с файлом такого размера — не мгновенный (загрузка + обработка на стороне провайдера). Нужен явный таймаут на запрос (например, 60–120с на чанк, не дефолтный таймаут HTTP-клиента) и ограниченное число ретраев с backoff на сетевые/5xx-ошибки (не ретраить 4xx — неверный формат/лимит превышен и с третьей попытки не станет валидным). План уже фиксирует, что ошибка транскрипции не блокирует воспроизведение файла (`Transcript.status: ERROR` независим от `MeetingFile.status: READY`) — это архитектурно верно и совпадает с рекомендацией здесь.
4. **Где именно в воркере.** Логично реализовать как отдельный шаг **после** успешного перехода `MeetingFile` в `READY` (Фаза 3 уже транскодировала и, что важно, скорее всего уже держала открытым ffmpeg-контекст для нужного файла) — отдельный `TranscriptWorkerService` (или расширение того же воркер-сервиса вторым `@Interval`), опрашивающий `Transcript.status = 'PENDING'` для `RECORDING`-файлов в статусе `READY`, точно тем же claim-паттерном (`updateMany` с условием), что и транскодирование в секции 3 — переиспользовать паттерн, не изобретать новый.

**Рекомендация:**
```
openai (официальный SDK, ^4.x/^5.x на момент 2025/2026 — если провайдер именно OpenAI Whisper;
если "аналог" — совместимый по форме audio/transcriptions REST-эндпоинт, тогда достаточно нативного fetch)
```
Извлекать audio-only дорожку перед отправкой (не слать видео), делить на ≤25MB чанки при необходимости, конкатенировать текст по порядку, явный таймаут + ограниченный retry с backoff только на сетевые/5xx, независимый статус `Transcript` от `MeetingFile.status`.

## 6. Prisma-моделирование: статус-жизненный цикл + polling

**Индекс на `status`.** Раз воркер каждый тик делает по сути `WHERE status = 'QUEUED' ORDER BY createdAt LIMIT 1` (или batched-версию), составной индекс `@@index([status, createdAt])` на `MeetingFile` — прямая рекомендация: при небольшом объёме строк (что здесь реалистично для MVP) это не критично для производительности, но это тривиальная и правильная практика, которая не будет мешать при росте, и явный сигнал в схеме, что таблица используется как очередь.

**Enum-моделирование** — консистентно с существующим `MeetingStatus` в `schema.prisma` (`enum` + `@default(...)` на колонке). Для `MeetingFile`:
```prisma
enum MeetingFileKind {
  RECORDING
  ATTACHMENT
}

enum MeetingFileStatus {
  QUEUED
  PROCESSING
  READY
  ERROR
}

model MeetingFile {
  id           String             @id @default(uuid())
  meetingId    String
  meeting      Meeting            @relation(fields: [meetingId], references: [id])
  uploaderId   String
  uploader     User               @relation(fields: [uploaderId], references: [id])
  kind         MeetingFileKind
  originalName String
  mimeType     String
  sizeBytes    Int
  storagePath  String
  processedPath String?
  status       MeetingFileStatus  @default(QUEUED)
  errorMessage String?
  createdAt    DateTime           @default(now())
  updatedAt    DateTime           @updatedAt

  transcript   Transcript?

  @@index([status, createdAt])
  @@map("meeting_files")
}
```
(псевдокод для иллюстрации структуры, не для копипаста без ревью относительно `onDelete`-семантики — `Meeting`/`User` в текущей схеме используют дефолтный `Restrict`, стоит явно решить при реализации, что происходит с `MeetingFile` при удалении `Meeting`/`User`, раз в PRD "удаление файлов" не в скоупе, но `Meeting`-модель, вероятно, теоретически удаляема).

**`Transcript` как отдельная модель 1:1 — план прав, стоит сохранить.** Обоснование: `Transcript` имеет собственный независимый жизненный цикл статусов (`PENDING/PROCESSING/READY/ERROR`), который эволюционирует асинхронно и с задержкой относительно `MeetingFile.status` (файл уже `READY`/воспроизводим, транскрипт может быть ещё `PENDING` или упасть в `ERROR` независимо) — хранить это как доп. колонки на `MeetingFile` означало бы либо дублировать четырёхзначный статус-enum на той же строке под другим именем (`transcriptStatus`), либо мешать в одну таблицу два независимых конечных автомата, что усложняет и запросы воркера (два разных `WHERE status = ...` на разных сущностях — чище, чем на одной), и будущее расширение (например, если транскрипт когда-то обзаведётся собственными полями вроде языка/провайдера/токенов). Отдельная модель с `meetingFileId String @unique` (1:1) — консистентна с тем, как в проекте уже разделены `User`/`Meeting`/будущий `MeetingFile` по ответственности модулей (`users` владеет `User`-доступом, `meetings` — `Meeting`, по аналогии новый `files`-модуль или отдельный `transcripts`-модуль владел бы `Transcript`). Единственный сценарий, где стоило бы сомневаться — если бы транскрипт всегда создавался синхронно с файлом и имел тот же статус-цикл, но это не так по плану (Фаза 5 явно делает его независимым).

**Рекомендация:** индекс `@@index([status, createdAt])` на `MeetingFile` (и аналогично на `Transcript`, раз у него свой воркер-опрос по статусу `PENDING`); enum-и по образцу `MeetingStatus`; `Transcript` — отдельная модель с `@unique` FK на `MeetingFile`, как и заложено в плане — веских причин сворачивать её в колонки нет.

## 7. Безопасность

**Хранение вне публично раздаваемой директории.** Ключевое: директория загрузок **не должна** попадать под `ServeStaticModule`/`express.static` или любой другой публичный root — единственный способ получить файл обратно должен быть через `GET /meetings/:id/files/:fileId/content`, защищённый `JwtAuthGuard` + `assertHostOrParticipant`. Это не только про приватность контента (файлы встречи не должны быть угадываемы по URL кем угодно), но и напрямую закрывает часть XSS-риска ниже — раз отдача идёт через контролируемый контроллер, а не статику, можно централизованно проставлять защитные заголовки на каждый ответ, а не полагаться на конфиг статик-сервера.

**Filename/path sanitization** — см. секцию 1: имя на диске всегда генерируется сервером (`randomUUID()`), пользовательское `originalName` никогда не участвует в построении файловой системы пути — это закрывает path traversal категорически, а не эвристиками вида "заменить `../`" (эвристики такого рода исторически обходились, например двойным URL-энкодингом или null-байтами — не полагаться на blacklist-санитайзинг, когда whitelist/генерация серверной стороной решает вопрос полностью).

**Zip-bomb / decompression** — подтверждено нерелевантно для этой фичи: ни на одном этапе плана сервер не распаковывает архивы (`ATTACHMENT` может быть, теоретически, `.zip`-файлом, но он хранится и отдаётся как есть, побайтово — не распаковывается на сервере). Единственное место, где сервер разбирает бинарный контент, — ffmpeg на video/audio (не архивный формат в этом смысле) и `file-type`, который читает только первые байты, не весь файл. Explicitly: если в будущем добавится, скажем, извлечение превью/метаданных из вложенных офисных документов (которые сами по себе zip-контейнеры — docx/pptx/xlsx), эта секция придётся пересмотреть — но текущий скоуп (просмотр офисных файлов не описан, только PDF/изображения через `<iframe>`/`<img>`) этого не требует.

**"upload a .php disguised as .mp4" — почему аллов-лист + content-sniffing из секции 1 закрывают именно этот класс проблем.** Классическая уязвимость этого класса — когда сервер (а) сохраняет файл под именем/расширением, которое веб-сервер/раннер интерпретирует как исполняемый код (например, Apache/PHP-FPM исполняет `.php` в докрут-директории), и/или (б) отдаёт файл с `Content-Type`, который заставляет **браузер** исполнить его как активный контент (HTML/SVG/JS). В этой архитектуре:

- (а) не применимо структурно: `apps/api` — Node/Nest-процесс, не Apache/PHP; файлы на диске никогда не выполняются как код никаким процессом в системе — они только читаются и стримятся как байты через `fs.createReadStream`. Расширение файла на диске не даёт исполняемости само по себе в этой среде.
- (б) — реальный и релевантный риск, но не через "почему .php", а через **браузер, который получит файл через `content`-эндпоинт**: если злоумышленник загрузит `.html`/`.svg`/`.xhtml`-файл под видом вложения, и сервер отдаст его с `Content-Type: text/html` или `image/svg+xml` **и без `X-Content-Type-Options: nosniff`**, браузер участника, открывшего этот "документ" внутри превью (`<iframe>`/`<embed>`, которые Фаза 4 явно вводит для не-медиа вложений), может исполнить встроенный `<script>` в контексте origin'а `apps/api` — classic stored XSS через file upload (документированный, распространённый класс, см. источники в исследовании). Это релевантно именно потому, что PRD включает превью произвольных вложений через `<iframe>`.

**Конкретные меры, которые закрывают (б):**
1. **`Content-Type` в ответе `content`-эндпоинта — брать из `file-type`-детекции (сохранённой в БД при загрузке), не из клиентского `mimetype`**, и **всегда** сопровождать заголовком `X-Content-Type-Options: nosniff` — это запрещает браузеру переопределять заявленный тип самостоятельной эвристикой (что и есть механизм, которым браузер иногда "спасает" неверно указанный тип, включая опасное автораспознавание HTML).
2. **MIME allow-list на загрузке должен исключать `text/html`, `image/svg+xml`, `application/xhtml+xml`, `application/javascript`/`text/javascript`** из списка допустимых `ATTACHMENT`-типов — раз PRD и так говорит про "документы, изображения" (не про HTML/SVG как таковые), явно исключить эти MIME из allow-list закрывает вопрос на входе, не только на выходе.
3. Для типов, которые всё равно допущены к inline-показу (PDF, растровые изображения — PNG/JPEG/GIF/WebP), нет активного контента в этом смысле (PDF теоретически может содержать JS, но `<iframe>` с PDF в современных браузерах рендерится PDF-вьювером браузера, а не выполняет PDF JS в контексте родительской страницы — отдельная, менее острая категория риска, не специфичная для этой фичи).
4. `Content-Disposition: inline` уместен для media/PDF/изображений (собственно "просмотр без скачивания" — прямое требование PRD), но раз "скачивание за пределы интерфейса" явно **не в скоупе** PRD, `inline` не противоречит этому — `inline` не то же самое, что "кнопка скачать", это просто то, как браузер рендерит ответ при прямом переходе по URL; сам факт, что эндпоинт требует `JwtAuthGuard`, уже ограничивает прямой доступ по URL без токена.

**Рекомендация:** хранение строго вне static root; серверные UUID-имена файлов на диске; allow-list MIME на загрузке явно исключает HTML/SVG/JS-типы; `content`-эндпоинт всегда отдаёт `Content-Type` из сохранённой при загрузке `file-type`-детекции + обязательный `X-Content-Type-Options: nosniff` на каждый ответ этого эндпоинта; zip-bomb — подтверждённо не в скоупе, ffmpeg/`file-type` читают файлы, не распаковывают контейнеры.

## Открытые риски / что стоит перепроверить при реализации

- **`fluent-ffmpeg` архивирован (май 2025).** Работает сегодня, но патчей от апстрима не будет; если попадётся exotic-кодек на входе, который сломает обёртку — план Б: прямой `child_process.spawn` на `ffmpeg-static`-бинарнике с ручной сборкой аргументов (несложно для одного фиксированного output-профиля, который здесь и нужен).
- **`ffmpeg-static` — сборка Docker-образа должна ставить зависимости на целевой платформе/архитектуре** (postinstall качает платформо-специфичный бинарь) — при мультистейдж/кросс-платформенной сборке легко получить бинарь не под ту ОС/архитектуру; стоит явно проверить на этапе Фазы 3, что `docker build` для прод-образа (если он появится) действительно тянет Linux-бинарь, а не кэширует Windows-артефакт с dev-машины.
- **`StreamableFile` + Range на iOS Safari — известный открытый баг Nest (`nestjs/nest#14873`)** без официального фикса на момент исследования; ручная реализация через `range-parser` снижает риск, но стоит явно протестировать воспроизведение аудио/видео с реального iPhone/iPad (не только через Chrome DevTools device emulation) перед тем, как считать Фазу 4 полностью готовой.
- **Whisper API 25MB-лимит и ценообразование могут измениться** — на момент исследования лимит стабилен с момента запуска API и не менялся, но точная стоимость по минуте/секунде — стоит свежо проверить у выбранного провайдера непосредственно перед реализацией Фазы 5 (не фиксировать здесь цифры, которые устареют быстрее, чем лимит файла).
- **Однопроцессное ограничение воркера** — осознанно принятый компромисс (нет очереди/Redis), но стоит явно документировать в `apps/api/CLAUDE.md` при реализации Фазы 3, чтобы будущий деплой с несколькими репликами `apps/api` не наступил на race condition молча.
- **`file-type` — ESM-only пакет**, подключение в CommonJS-проекте (текущий `apps/api` компилируется через `nest build`/`tsc` в CommonJS) требует динамического `import()`, а не `require`/статического `import` — стоит проверить конкретный паттерн подключения на актуальной версии пакета непосредственно при реализации, ESM/CJS-взаимодействие имеет свойство меняться между мажорными версиями таких пакетов.
- **`@nestjs/schedule` не в текущих зависимостях `apps/api`** — версия должна быть подобрана под Nest 11 на момент реализации (совместимость стоит проверить в `package.json`/README пакета, а не полагаться на цифру версии из этого документа, которая могла устареть).
